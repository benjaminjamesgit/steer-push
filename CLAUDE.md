# steer-push — working conventions

Mission: test whether a criterion exists that distinguishes when a physical difference STEERS a system (selection) from when it merely PUSHES it (causation) — BELOW life: no chooser, no self-maintaining beneficiary, no observer. Empirical arm: toy dynamical systems judged against the fixed adversarial bar in pre_registration.md.

Discipline:
- Pre-register before implementing. Criterion, toy, measures, and predicted verdicts go into pre_registration.md (append-only) BEFORE any run. Never move the bar after seeing results.
- Default to RENAMING. A result is a contribution only if it does work the existing criteria (Friston FEP, Deacon teleodynamics, Bateson, Pattee, assembly theory) demonstrably do not. Flag any step that reduces to one of them.
- Matched controls. Every STEER candidate is tested against a matched PUSH control: two systems identical except for the steer/push feature. A criterion that cannot separate a hand-built pair is not a criterion.
- Observer-free / chooser-free. Every operational definition must be computable from the system's own states and dynamics — no observer, no "as-if" interpretation, no presupposed code, measurement apparatus, or beneficiary.
- Surgical edits; ASCII in code, Unicode only in prose docs; commits via two -m flags with a dense body; version strings "vX.Y.Z: subject".
